Capstone Project · Jan 2026 – Apr 2026

Conversational AI for Emergency Checklists

React · Supabase · OpenAI · Claude API · Runway AI · Python

Overview

Built for Innomergence Solutions, an emergency management company serving First Nations communities and local governments. The system converts static emergency checklists into structured, machine-readable primitives and produces 60-second training videos through a guided AI-assisted workflow. It was built as a 4-person capstone team project.

The Problem

Emergency Operations Centres depend on static PDFs and Word documents for decision-making during disasters. Under pressure, manual reading causes inconsistencies and delays. There was no structured, machine-readable way to represent checklist logic or automate any part of the process.

Architecture

React frontend for user interaction and workflow state. Supabase handles authentication, database, file storage, and serverless edge functions that run all AI operations. Separating AI work into edge functions allows those operations to scale independently without affecting the rest of the application.

Key Features

  • Primitive extraction converts checklist items into structured fields: Who, Trigger Condition, Preconditions, Required Action, Verification Method, and Failure Consequences.
  • Dual-model script generation where OpenAI and Claude each produce a script independently, then cross-validate each other with confidence scores against the original document.
  • Conversational editing lets users refine primitives using natural language or voice input through the browser Speech API, handled by the smart-action edge function.
  • Monte Carlo reliability simulation assigns weights to critical fields and runs probabilistic risk scoring rather than a simple pass/fail check.
  • Runway AI media pipeline converts approved scripts into synchronized 60-second training videos with voice narration through a custom SyncedPlayer component.

From Prototype to Production

Before building the full app, a Python CLI prototype was built using pdfplumber and Streamlit to validate that LLMs could reliably extract structured rules from emergency documents. Key lessons around strict JSON output enforcement, prompt temperature tuning, and page-boundary handling were carried directly into the production edge functions.

Security Tool · Jan 2026 – Present

SQLi Scout

Python · PyQt5 · Docker · PyInstaller

Overview

A zero-install SQL injection vulnerability scanner with a full GUI. Built to make professional-grade security testing accessible to developers without a security background. Ships as a standalone Windows executable compiled with PyInstaller, requiring no Python installation on the target machine. Validated against OWASP Juice Shop 17.1.1 with 78 vulnerabilities confirmed and zero false positives.

The Problem

SQL injection has been a top-ranked web vulnerability for over two decades, yet most deployed applications go untested because the available tools require either a paid license, terminal proficiency, or hands-on security knowledge that small development teams do not have. Burp Suite costs around $449 USD per year and assumes familiarity with HTTP proxy interception. sqlmap is highly capable but requires manual parameter configuration from the command line. There was no lightweight, guided option for developers who just need to know if their application is exposed.

Architecture

SQLi Scout is organized into six Python modules: crawler.py for endpoint discovery, scanner.py for injection detection, payloads.py for the payload and signature library, reporter.py for CLI output formatting, utils.py for shared helper functions, and gui.py for the PyQt5 graphical interface. The scanning backend is entirely independent of the GUI layer, meaning the tool can also be driven as a command-line application. The final deliverable is compiled using PyInstaller into a single Windows executable with all dependencies bundled, keeping the output binary below 100 MB.

Key Features

  • Four detection methods: error-based detection using 36 base payloads and 28 database error signatures, boolean-based blind detection using 12 true/false payload pairs evaluated with a difflib similarity heuristic, time-based blind detection using 27 delay payloads across five database dialects, and union-based data extraction using dynamic column-count probing.
  • Endpoint discovery runs in two phases. The first silently probes a universal list of 48 GET and 24 POST paths common across web applications. The second recursively follows internal HTML links, parsing forms, extracting URL parameters, and scanning inline JavaScript for API paths using four regex patterns. All results are deduplicated before being passed to the scanner.
  • WAF evasion is applied by default to the first six error-based payloads, generating variants through inline comment injection, alternating case randomization, URL encoding, double URL encoding, whitespace substitution, hex encoding, and a null-byte prefix, producing up to 48 unique payloads per scan run.
  • Union-based extraction runs a three-step process: ORDER BY escalation to determine the column count, marker injection to identify the reflected column index, and a set of pre-built extraction queries targeting SQLite to retrieve version strings, table names, and credential dumps from the Users table.
  • The PyQt5 GUI exposes full scan configuration through a sidebar, displays findings in a live-updating vulnerability table with severity badges, and includes a separate Extracted Data tab for credential dumps. Results can be exported to JSON at any time.

Validation Results

A full-depth scan of OWASP Juice Shop 17.1.1 ran for 30 minutes and 31 seconds, testing 68 endpoints and 99 parameters across all four detection methods. The scanner confirmed 78 vulnerabilities: 75 rated CRITICAL through union-based extraction, 2 rated HIGH through error-based and boolean-based blind detection on the products search and login endpoints, and 1 rated MEDIUM through a time-based timeout. No false positives were recorded across any of the 99 parameters tested.

Limitations and Next Steps

The static crawler does not execute JavaScript, so endpoints loaded dynamically by single-page application frameworks are not discoverable without a headless browser integration. The union extraction queries are currently hardcoded for SQLite and would need to be extended for MySQL, PostgreSQL, and MSSQL. The boolean-based heuristic uses fixed thresholds that can produce false negatives when applications return structurally similar error and success responses. Future development priorities include Playwright integration for JavaScript-rendered endpoints, session management support for authenticated scanning, and a PDF report export feature.

COMP 4610 · Advanced Database Systems · Jan 2026 – Apr 2026

Employment Outcomes Predictor

Python · scikit-learn · pandas · SMOTE · imbalanced-learn · matplotlib

Overview

A supervised classification project built on the January 2026 Statistics Canada Labour Force Survey (LFS) Public Use Microdata File. The goal was to predict whether young Canadians aged 15 to 34 who are no longer in school are currently employed, using only demographic and educational variables. Starting from 112,498 raw survey records, 19,666 were retained after population filtering for the target group.

The Problem

The raw dataset had an 80/20 class split between employed and not-employed individuals. A naive classifier trained on this data achieves 81% accuracy by predicting employed for nearly every record, but only identifies 15% of at-risk individuals. That makes it useless for the actual goal of identifying people who need employment support. Three class imbalance strategies were compared to fix this.

Data Pipeline

  • All 80 LFS variables are numeric codes requiring manual mapping from the Statistics Canada codebook before any analysis could run.
  • UNION (union membership) was excluded as a feature because it is only collected for employed respondents, which would cause the model to detect unemployment through missing values rather than genuine demographic signals.
  • AGE_6 nulls for the 30-34 age group were imputed with 0 rather than dropped, preserving 40.6% of the filtered population that would otherwise be lost.
  • Four categorical variables (PROV, CMA, MARSTAT, IMMIG) were one-hot encoded, expanding the feature matrix from 8 raw variables to 33 binary and ordinal features with zero nulls.

Class Imbalance Handling

  • Default (no adjustment): 81% accuracy but only 15% recall on not-employed. Useless for targeting at-risk individuals.
  • Balanced class weights: Not-employed recall improved to 68% but overall accuracy dropped to 67% and employed recall fell too far.
  • SMOTE: Generated synthetic minority samples on the training set only, expanding it from 15,732 to 25,358 records at a 50/50 balance. Test set was kept at its real 80/20 distribution. Best overall trade-off: 59% not-employed recall, 75% employed recall, 72% accuracy, F1 of 0.794.

Model Results

  • Decision Tree with SMOTE at depth 3 was the best model: 72% accuracy, 59% not-employed recall, cross-validated F1 of 0.794.
  • Random Forest with 100 trees and SMOTE scored lower on every metric: 68% accuracy, 48% not-employed recall, F1 of 0.789. The single tree outperformed the ensemble because education level (EDUC) accounts for 92.5% of the Decision Tree's feature importance. The Random Forest's random feature subsampling forces it to train trees that sometimes exclude EDUC entirely, adding noise without reducing bias.
  • 5-fold cross-validation confirmed results were stable across splits, with DT + SMOTE achieving a mean F1 of 0.794 and standard deviation of 0.010.

Key Findings

  • Education level is the single strongest predictor, accounting for 92.5% of feature importance in the Decision Tree. Individuals with no credential or some high school and not in Quebec have an 83.3% predicted probability of being not employed.
  • Immigration status appears as a significant independent predictor across both models, meaning employment barriers for immigrants are not fully explained by education level, age, or other demographic factors in the survey.
  • The Random Forest's more distributed feature importance profile (EDUC 31.2%, AGE_6 8.7%, GENDER 7.8%) gives a more complete picture of employment as a multifactorial outcome.

Machine Learning · Sep 2025 – Dec 2025

League of Legends Win Predictor

Python · scikit-learn · pandas

Overview

A multi-model machine learning project trained on 80,000 ranked League of Legends matches to predict game outcomes from player and team statistics. Built to move beyond raw accuracy and uncover non-obvious strategic insights from large-scale gameplay data collected via the Riot Games API.

Data and Approach

The original dataset contained over 180,000 matches across multiple CSV files (730 MB total), including player stats, match metadata, objectives, and champion selections. After preprocessing and filtering (remakes, AFK players, and season constraints), the dataset was reduced to 80,000 matches for computational efficiency. Feature relationships were explored using correlation heatmaps, pairplots, and PCA, which showed that two principal components alone captured over 99% of the variance, indicating strong feature interdependence.

Models and Results

  • Trained and tuned Random Forest, AdaBoost, Gradient Boosting, and MLP models using scikit-learn.
  • Applied feature scaling, estimator tuning, and architecture adjustments to optimize performance across models.
  • Gradient Boosting achieved the highest performance with 87.68% accuracy and 0.8754 F1 score, with strong cross-validation stability (~0.88).
  • All models significantly outperformed the baseline (50%), confirming predictive signal in gameplay statistics.

Key Findings

  • Deaths and assists were stronger predictors of winning than kills, challenging common player assumptions about performance.
  • Feature importance and decision tree analysis highlighted survivability, team coordination, inhibitor control, and gold advantage as primary win drivers.
  • PCA and correlation analysis revealed that many gameplay metrics (kills, gold, damage) are highly correlated and contribute overlapping information.

Insights

The results emphasize that League of Legends is fundamentally a team-oriented and strategy-driven game. Models consistently showed that minimizing mistakes (deaths) and enabling team play (assists and objectives) are more impactful than individual kill counts, providing data-backed insight into optimal gameplay strategies.
